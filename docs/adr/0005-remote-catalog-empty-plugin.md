# ADR 0005: Empty plugin, remote catalog

- **Status:** Accepted
- **Date:** 2026-04-27
- **Context:** Before the public 1.0.0 baseline, ODD shipped every wallpaper scene, icon set, and widget as bundled plugin content. The resulting `odd.zip` was pushing 1 MB with ~19 scenes, 17 icon sets, 2 widgets, and an internal app catalog with seven `.wp` archives. Adding a single new wallpaper meant cutting a plugin release; every install had to carry every scene ever shipped whether the user wanted it or not. The `odd/bin/validate-scenes` + `validate-icon-sets` + app-catalog validators ran per-subsystem and had diverged in their rigor. Meanwhile the universal `.wp` format (ADR 0002) already existed and was used for admin uploads — we had a working installer for every content type that nobody had hooked up to a remote source.
- **Decision:** The plugin ships empty. Every piece of user-facing content lives in a remote catalog published at `https://odd.regionallyfamous.com/catalog/v1/registry.json` (a GitHub Pages deploy of this repo's `site/catalog/v1/` tree). On activation, ODD reads the registry's `starter_pack` block inline, downloads a small default scene + icon set through the existing `oddout_bundle_install()` pipeline, and verifies each bundle's SHA256 against the registry before extraction. A privileged `init` safety net repeats the same synchronous install path when the site is temporarily offline or locked, with backoff stored in `oddout_starter_state`; it does **not** rely on WP-Cron. Every other piece of content installs the same way, on demand, from the ODD Shop. The remote catalog is authored in `_tools/catalog-sources/` and built by `_tools/build-catalog.py` — a deterministic builder whose output is byte-for-byte stable across runs. `odd/bin/validate-catalog` enforces schema, size, SHA256, and starter-pack resolution on every CI run, and the `install-smoke` workflow intercepts HTTP calls with an MU-plugin (`ci/smoke/odd-smoke-fixture.php`) so the starter pack can be tested hermetically without depending on the live Pages deploy.
- **Consequences:**
  - The plugin zip drops from ~1 MB to <500 KB. The `build-zip` budget drops from the old oversized allowance to 2 MB so bundled content regressions fail CI.
  - Shipping new content no longer requires a plugin release. A PR that drops a new scene under `_tools/catalog-sources/scenes/<slug>/` and lands on `main` publishes to the catalog on the next `pages.yml` deploy — ~2 minutes.
  - Fresh installs may see an empty Shop for a few seconds while the starter pack resolves. A built-in `odd-pending` gradient scene paints in the meantime so the desktop is never blank. The starter-pack runner retries inline with exponential backoff if the catalog is unreachable, so temporary outages are transparent without depending on loopback cron.
  - Downloaded bundles are verified against a registry-declared SHA256 before extraction. Tampered bundles can't take the install over even if the CDN is compromised.
  - The catalog and install surface consolidates on `/odd/v1/bundles/*`. App-specific routes remain only for installed app runtime operations such as list, open metadata, toggle surfaces, serve files, icons, upload, and delete.
  - Enterprise deployments that can't reach `odd.regionallyfamous.com` can point at their own mirror via `define( 'ODDOUT_CATALOG_URL', … )` or the `oddout_catalog_url` filter.
  - `odd/bin/validate-scenes` and `odd/bin/validate-icon-sets` are deleted and folded into `odd/bin/validate-catalog`.
- **Alternatives considered:**
  - *Keep everything bundled.* Rejected: ties content velocity to plugin release cadence. Users who want "one more scene" have to wait for the next release + plugin update flow.
  - *Bundle a starter pack, fetch the rest remotely.* Rejected: halves the win. The bundled content still has to go through CI validation and eats zip budget. Shipping a single 30 KB `odd-pending` fallback scene and pulling the starter itself from the catalog keeps the plugin genuinely empty.
  - *Put the catalog in a separate repo.* Rejected: the bundle source (`_tools/catalog-sources/`), the builder, and the validator live next to the PHP that consumes them. Splitting the repo introduces cross-repo PR coordination for every scene addition and makes the CI determinism check harder to run. Both the plugin and the catalog deploy from this repo; the Pages workflow publishes `site/` (catalog + marketing), and the release workflow publishes `odd.zip` (plugin only).
  - *Host bundles as GitHub Releases assets instead of Pages.* Rejected: GitHub Releases occasionally 409s on asset uploads (the release workflow has retry logic for exactly this), and the release page's asset CDN carries less predictable CORS behavior than GitHub Pages. Pages has shipped `access-control-allow-origin: *` for years; Releases doesn't make the same guarantee.
  - *Ship content as individual WordPress plugins, installable through wp.org.* Rejected: the wp.org review turnaround is days, not seconds. This plugin's own review cycle is already a bottleneck; multiplying it by the number of content bundles is untenable for a project that wants new scenes to land in hours.
